# Lakehouse Platform

Lakehouse assets are split into Bronze, Silver and Gold layers.

## Layer Rules

- Bronze: raw immutable events and CDC changes. Minimal parsing, full replay value.
- Silver: conformed, deduplicated, typed and PII-classified data.
- Gold: business-ready marts, metrics and ML feature datasets.

## Production Requirements

- Use event-time partitioning and preserve ingestion-time metadata.
- Record source offset/snapshot for every row lineage.
- Apply the Iceberg naming, partitioning and table maintenance standard in
  [iceberg-table-standards.md](iceberg-table-standards.md).
- Gate production-like Bronze commits through [bronze-lakehouse-ops.md](bronze-lakehouse-ops.md)
  and `enterprise-dp bronze-lakehouse-ops-report`, which checks offset ledgers, replay proof,
  Iceberg commit metadata, append-only enforcement and table maintenance evidence.
- All Gold writes must be atomic and quality-gated.
- Production-like Silver/Gold commits must emit `lakehouse_snapshot_evidence.v1`, generated by
  `enterprise-dp snapshot-evidence-record`, so release gates can verify Iceberg snapshot metadata,
  schema/partition hashes, row counts, content hashes and upstream offset ledger binding.
- Gate production-like Silver/Gold exposure through
  [silver-gold-publication-ops.md](silver-gold-publication-ops.md) and
  `enterprise-dp silver-gold-publication-ops-report`, which checks the release evidence,
  promotion manifest, activation manifest, active pointer, rollback target and production change
  ticket before Control Tower can sign off.
- Gate production-like Iceberg catalog runtime readiness through
  [catalog-runtime-ops.md](catalog-runtime-ops.md) and
  `enterprise-dp catalog-runtime-ops-report`. This is the only lakehouse evidence that can close
  production catalog HA, multi-AZ failover, stale-commit rejection, backup/restore/PITR and catalog
  audit gaps in the production review pack.
- Backfills must be reproducible from Bronze snapshots.

## Implemented Local Evidence

- `enterprise-dp iceberg-catalog-smoke` commits the finance Bronze/Silver/Gold local Parquet
  outputs into a PyIceberg SQL catalog, verifies Iceberg snapshot metadata files and scans each table
  back. This produces `iceberg_catalog_smoke_report.v1` for the production review pack.
- `enterprise-dp trino-iceberg-minio-smoke` creates and queries a finance Gold Iceberg table in a
  MinIO-backed warehouse through Trino and a local Postgres JDBC catalog. It verifies Trino query
  results, Iceberg `$snapshots`, Iceberg `$files` and MinIO data/metadata objects.
- `enterprise-dp catalog-cross-engine-smoke` verifies the same local Postgres JDBC catalog is usable
  from both Trino and PyIceberg: Trino creates the table, PyIceberg reads and appends, Trino reads
  the PyIceberg commit and a stale PyIceberg commit is rejected.
- `enterprise-dp live-bronze-ingestion-smoke` writes the finance benefit outbox events from a live
  local Redpanda round-trip into a Bronze Iceberg table on MinIO, verifies approved row count,
  duplicate/idempotency handling, quarantine/DLT evidence, offset-ledger restart resume, Iceberg
  snapshots/files and Trino read-back.
- `enterprise-dp orchestrated-publication-smoke` reads the live Bronze finance Iceberg table,
  builds Silver and Gold finance reconciliation outputs, writes both into the local MinIO-backed
  Iceberg warehouse, verifies both through Trino, and emits release evidence, promotion,
  activation, rollback pointer and active-pointer drift-negative evidence from a local Dagster
  in-process run.
- `enterprise-dp trino-runtime-security-smoke` verifies local Trino file-based authorization over the
  same Iceberg/MinIO Gold table with explicit allow, write-deny, select-deny and default-deny
  probes.
- These are local runtime proofs. They are not yet production catalog HA/concurrency,
  production managed failover, backup/restore/PITR, production Dagster daemon/distributed executor,
  production retry/backfill policy, cloud KMS/bucket-policy attestation, production authentication,
  masking or audit proof. Those catalog runtime gaps require an attached staging/prod
  `catalog_runtime_ops_report.v1`.
