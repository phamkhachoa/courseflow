CourseFlow Roadmap: hướng dẫn phát triển từng bước

Tài liệu này chuyển mô tả tổng quát của CourseFlow thành các bài làm cụ thể. Người làm đi từ việc chạy được hệ thống, đọc đúng service, thêm API nhỏ, đến tự thiết kế một luồng nghiệp vụ sẵn sàng chạy production có gateway, BFF, database, event, search, analytics, web/app và test.

1. Cách đọc lộ trình

Lộ trình chia theo năng lực, không chia cứng theo số tuần. Mỗi bước đều có sản phẩm cụ thể, cách kiểm chứng và tiêu chí review. Nếu một bước chưa chắc, hãy lặp lại bằng một service khác trước khi đi tiếp.

Mốc Dấu hiệu đạt Bài làm phù hợp Cách review
Nền tảng Đọc hiểu code có mẫu, chạy được project, thêm/sửa endpoint nhỏ, biết báo cáo lỗi bằng log và request. CRUD nhỏ, DTO, validation, migration đơn giản, sửa bug rõ nguyên nhân. Review theo checklist, nhắc convention, kiểm tra command/curl.
Làm tính năng Tự hoàn thành một phần tính năng end-to-end, tự viết test, biết transaction và edge case cơ bản. Một tính năng trong một service, consumer event đơn giản, màn hình admin/người học nhỏ. Review thiết kế, bảo mật, idempotency, test gap.
Production Tự thiết kế tính năng đi qua nhiều service, cân nhắc consistency, event, BFF, observability, migration và rollout. Luồng nghiệp vụ nhiều service, BFF aggregation, search/analytics read model, gia cố production. Review trade-off, compatibility, rollout plan và rủi ro vận hành.

2. Nguyên tắc làm việc trong dự án

Đọc trước khi code

Luôn đọc một service đã có pattern tương tự trước khi thêm code. Ví dụ muốn làm CRUD thì đọc media-service hoặc enrollment-service; muốn event thì đọc announcement-service.

Một PR, một mục tiêu

Mỗi PR chỉ nên có một mục tiêu rõ ràng: migration + DTO + repository + service + controller + test. Không trộn refactor, đổi UI và sửa infra trong cùng PR.

Không đoán mò

Khi lỗi, ghi lại request, response, log, stack trace, câu SQL hoặc payload event. Tự hỏi lỗi nằm ở boundary nào: gateway, controller, service, DB, Kafka hay frontend.

Done là chạy được

Done không phải là "đã viết code". Done là build pass, test pass, có data demo hoặc curl để kiểm chứng, tài liệu/API được cập nhật nếu thay đổi contract.

Khi bị kẹt, hãy ghi 4 dòng trước khi hỏi: mục tiêu đang làm, lỗi đang gặp, đã thử gì, nghi ngờ hiện tại là gì. Cách này giúp người review hiểu vấn đề nhanh và giúp bạn tự tìm ra lỗi tốt hơn.

4. Từ bài toán tổng quát đến việc làm thực tế

CourseFlow là hệ thống LMS nhiều module. Khi nhận một yêu cầu, đừng bắt đầu bằng cách tìm chỗ sửa ngẫu nhiên. Hãy xác định trước: người dùng là ai, service nào sở hữu nghiệp vụ, dữ liệu nằm ở đâu, client nào cần màn hình, và có cần event/read model hay không.

Bài toán sản phẩm Service chính Client liên quan Việc nên làm trước
Danh mục khóa học và lộ trình học course-service Next.js, Flutter, React admin Tạo/sửa course module, public course detail, tiến độ học theo module.
Ghi danh và danh sách lớp enrollment-service, organization-service React admin, Flutter Ghi danh người học, kiểm tra capacity, chuyển sang waitlist khi lớp đầy.
Bài tập, hạn nộp và nhắc hạn assignment-service, deadline-service, notification-service Flutter, React admin Tạo assignment, sinh deadline reminder, gửi notification qua event.
Thông báo và thảo luận lớp học announcement-service, discussion-service, notification-service Next.js, Flutter, React admin Publish announcement, tạo discussion thread, moderation và accepted answer.
Quiz, sổ điểm và đánh giá đồng cấp quiz-service, gradebook-service, peer-review-service Flutter, React admin Submit quiz attempt, auto-grade, sync gradebook, phân công peer review.
Chứng chỉ, tìm kiếm và phân tích certificate-service, search-service, analytics-service Next.js, React admin Xác minh chứng chỉ public, index course vào Elasticsearch, dashboard cảnh báo rủi ro.

Cách bóc một yêu cầu mới

  1. Viết lại yêu cầu bằng một câu: actor nào làm gì để đạt kết quả gì.
  2. Chọn service sở hữu nghiệp vụ. Nếu cần nhiều service, xác định service nào là source of truth.
  3. Liệt kê dữ liệu cần lưu, dữ liệu chỉ để hiển thị, dữ liệu đến từ event hoặc BFF.
  4. Thiết kế API command/query trước, sau đó mới viết migration và code.
  5. Chọn cách kiểm chứng: test, curl, màn hình UI, hoặc event replay.
Ví dụ yêu cầu "nhắc học viên sắp trễ hạn" không chỉ là một API gửi thông báo. Nó gồm assignment/deadline làm source of truth, outbox event để báo reminder due, notification-service tạo inbox item, analytics-service cập nhật risk signal và React admin hiển thị dashboard.

5. Tiêu chuẩn chất lượng cho mọi thay đổi

  • Không expose entity/model trực tiếp ra API; chỉ trả DTO.
  • Không service nào query database của service khác.
  • Migration production chỉ tạo schema; demo data nằm ở profile demo.
  • Controller validate input bằng annotation và @Valid.
  • Service chứa use case/transaction; repository chỉ làm data access.
  • Nếu ghi event, event phải nằm trong cùng transaction với thay đổi domain.
  • Nếu consume event, phải có dedup bằng processed_events.
  • Gateway là nơi duy nhất tin JWT từ client; downstream chỉ tin header do gateway inject.
  • Mỗi thay đổi có test hoặc ít nhất command/curl chứng minh chạy được.

cd ../react-admin

Kết quả cần có

  • Ghi chú lỗi setup gặp phải và cách xử lý.

Bước 1 - Đọc code theo luồng request Nền tảng

Mục tiêu: hiểu một request đi qua những lớp nào, không còn nhìn project như một đống folder rời rạc.

Đọc theo thứ tự

  1. backend/docs/architecture/backend-architecture.md: hiểu service ownership.
  2. services/api-gateway/application.yml: hiểu route public/internal.
  3. services/course-service: đọc catalog + module APIs.
  4. services/identity-service: đọc login, JWT, refresh token.
  5. common-library: đọc error, CurrentUser, GatewayHeaders.

Bài tập

  • Vẽ sơ đồ request GET /api/public/courses từ client tới DB.
  • Vẽ sơ đồ request POST /api/auth/login.
  • Giải thích vì sao client không được tự gửi X-User-Id.

Kết quả cần đạt

  • Trả lời được controller/service/repository/dto/model khác nhau thế nào.
  • Biết file nào định nghĩa route gateway.
  • Biết service nào dùng PostgreSQL, MongoDB, Elasticsearch.

Bước 2 - CRUD đầu tiên Nền tảng

Mục tiêu: thêm một API nhỏ theo pattern có sẵn, không phá kiến trúc.

Bài làm mẫu: thêm endpoint list theo filter trong media-service

  1. Đọc media-service/controller/MediaController.java.
  2. Đọc DTO trong MediaDtos.java.
  3. Đọc SQL trong MediaRepository.java.
  4. Thêm filter mới, ví dụ contentType.
  5. Chạy compile và test.
GET /internal/media/assets?ownerId=2&contentType=application/pdf
Không thêm field vào entity/model nếu DB chưa có column. Nếu cần column mới, phải viết migration.

Kết quả cần đạt

  • API trả đúng DTO.
  • Không thay đổi business service khác.
  • mvn -pl services/media-service -am test pass.
  • Có curl minh họa request/response.

Bước 3 - API, DTO, validation và error Nền tảng

Mục tiêu: hiểu API contract và biết bảo vệ input trước khi vào business logic.

Bài làm mẫu: tạo request DTO đúng chuẩn

  • Dùng record DTO cho request/response.
  • Thêm annotation như @NotBlank, @NotNull, @Positive.
  • Controller nhận @Valid @RequestBody.
  • Không trả raw map nếu có thể tạo DTO rõ ràng.

Bài tập

Thêm endpoint tạo notification preference nếu chưa có, hoặc mở rộng preference theo channel.

POST /internal/notifications/preferences
{
  "userId": "4",
  "channel": "IN_APP",
  "enabled": true
}

Kết quả cần đạt

  • Request thiếu field trả 400 qua ApiExceptionHandler.
  • Response không lộ column/field thừa.
  • Tên DTO kết thúc bằng Dto, không dùng Vm.

Bước 4 - Database và Liquibase Nền tảng

Mục tiêu: biết thay đổi schema an toàn và không lẫn demo data vào production migration.

Quy tắc migration

  • db.changelog-master.yaml chỉ include schema.
  • db.changelog-demo.yaml include schema + demo data.
  • Không viết INSERT demo vào file schema production.
  • Không xóa/đổi changeset đã chạy; thêm changeset mới.

Bài làm mẫu

Thêm column description cho một bảng nhỏ, ví dụ notification_preferences hoặc media_assets.

-- liquibase formatted sql
-- changeset courseflow:media-002-add-description
ALTER TABLE media_assets ADD COLUMN IF NOT EXISTS description TEXT;

Kết quả cần đạt

  • Migration chạy với DB sạch.
  • Code compile sau khi thêm field.
  • Demo data nếu có phải ở file 900-demo-data.sql hoặc changeset demo riêng.

Bước 5 - Xây một tính năng trong service Mở rộng

Mục tiêu: tự làm một tính năng trong một service từ DB đến API.

Chọn một bài làm

Bài làm Service Giá trị học được
Course section capacity organization-service / enrollment-service Transaction, capacity, waitlist.
Announcement scheduling announcement-service Status lifecycle, publish time, outbox.
Discussion moderation discussion-service State change, accepted comment, audit thinking.
Media signed upload policy media-service Storage abstraction, metadata, security boundary.

Trình tự triển khai

  1. Viết use case bằng lời: actor, input, output, rule.
  2. Thiết kế DTO request/response.
  3. Thêm migration nếu cần.
  4. Viết repository SQL.
  5. Viết service transaction.
  6. Viết controller.
  7. Viết test hoặc curl verify.
  8. Cập nhật API doc nếu contract mới.

Kết quả cần đạt

  • Người review có thể gọi API mà không cần đọc code trước.
  • Edge case chính được xử lý: not found, conflict, invalid input.
  • Không có logic nghiệp vụ trong controller.

Bước 6 - Auth, gateway và CurrentUser Mở rộng

Mục tiêu: hiểu trust boundary và không tạo lỗ hổng bằng cách tin header từ client.

Đọc kỹ

  • JwtAuthenticationGatewayFilter.java
  • GatewayHeaders.java
  • CurrentUserArgumentResolver.java
  • JwtAuthenticationGatewayFilterTest.java

Bài tập

Thêm một endpoint chỉ operator được gọi, ví dụ publish announcement hoặc archive course, dùng CurrentUser để lấy actor.

POST /internal/courses/{courseId}/archive
Authorization: Bearer <token>

Kết quả cần đạt

  • Endpoint không đọc X-User-Id thủ công.
  • Token của người học không gọi được route backoffice/user-management.
  • Gateway test vẫn pass.

Bước 7 - Transactional outbox Mở rộng

Mục tiêu: hiểu vì sao không publish Kafka trực tiếp trong cùng method ghi DB.

Luồng mẫu

Domain action
  ├─ update domain table
  └─ insert outbox_events
        same DB transaction

Relay/Debezium
  └─ publish Kafka later

Consumer
  └─ process event with dedup

Bài tập

Chọn một producer đã có outbox, ví dụ assignment-service với submission.created, rồi chuẩn hóa payload có đủ eventId, aggregateId, occurredAt, actorId nếu cần.

Outbox payload phải đủ để consumer xử lý mà không query database producer. Nếu thiếu dữ liệu, consumer sẽ buộc gọi ngược service hoặc đọc DB sai nguyên tắc.

Kết quả cần đạt

  • Domain change và outbox insert nằm cùng transaction.
  • Payload có event id duy nhất để dedup.
  • Không publish Kafka trực tiếp từ service method nghiệp vụ.

Bước 8 - Consumer và dedup Mở rộng

Mục tiêu: viết consumer an toàn với Kafka at-least-once delivery.

Đọc luồng mẫu

  • announcement-service/config/OutboxRelay.java nếu có relay mẫu trong repo.
  • notification-service/consumer/AnnouncementEventConsumer.java nếu có consumer mẫu.
  • Bảng processed_events trong changelog consumer service.

Bài tập đề xuất

Producer event Consumer cần viết Kết quả
deadline.reminder.due notification-service Tạo notification reminder.
discussion.comment.created analytics-service Tăng discussion count.
quiz.attempt.graded gradebook-service Upsert grade entry.

Kết quả cần đạt

  • Consumer check processed_events trước khi xử lý.
  • Insert domain/read-model update và insert processed event cùng transaction.
  • Gửi lại cùng event 2 lần không tạo dữ liệu trùng.

Bước 10 - Search và analytics read model Production

Mục tiêu: hiểu hệ thống đọc nhanh bằng read model, không query chéo DB service khác.

Search

  1. Course-service publish hoặc thay đổi course.
  2. Event/CDC đưa dữ liệu sang search-service.
  3. Search-service index Elasticsearch.
  4. Next.js gọi /api/public/search/courses?q=....

Analytics

  1. Consumer nhận event từ enrollment, submission, discussion, quiz, certificate.
  2. Update bảng metrics riêng của analytics-service.
  3. React admin gọi dashboard từ analytics API.

Kết quả cần đạt

  • Search không đọc DB course-service trực tiếp.
  • Analytics update idempotent khi event bị gửi lại.
  • Có cách rebuild/replay read model từ event hoặc CDC.

Bước 11 - BFF composition Production

Mục tiêu: biết khi nào cần BFF và không biến gateway thành BFF trá hình.

Trường hợp nên làm BFF

BFF endpoint Gọi service nào Client dùng
GET /api/learning/me/dashboard course, enrollment, deadline, notification, gradebook Flutter, Next cho người học đã đăng nhập.
GET /api/backoffice/courses/{id}/operations course, enrollment, announcement, discussion, analytics React admin.
GET /api/backoffice/students/{id}/risk analytics, gradebook, quiz, deadline React admin/advisor.

Quy tắc

  • Gateway route/auth/rate-limit; BFF compose data; domain service giữ business rule.
  • BFF không sở hữu DB nghiệp vụ.
  • BFF DTO được thiết kế theo màn hình, không mirror DTO của từng service y nguyên.

Kết quả cần đạt

  • Một màn hình client gọi một BFF endpoint thay vì gọi 5 service.
  • BFF có timeout/fallback rõ ràng cho downstream.
  • BFF không mutate dữ liệu ngoài việc gọi command API của domain service.

Bước 14 - Gia cố production Production

Mục tiêu: nhìn hệ thống bằng con mắt vận hành: fail ở đâu, quan sát thế nào, rollback ra sao.

Observability

  • Correlation id end-to-end.
  • Structured logs.
  • Metrics: latency, error rate, consumer lag.

Reliability

  • Timeout/retry/circuit breaker cho service call.
  • Outbox retry và dead-letter topic.
  • Idempotency key cho command nguy hiểm.

Security

  • JWT secret không hardcode prod.
  • Role/permission theo use case.
  • Không log token/password/PII.

Release

  • Backward-compatible migration.
  • Feature flag nếu thay đổi lớn.
  • Rollback plan cho schema/API/event.

Rubric tự đánh giá năng lực

Bài tổng hợp production

Đây là bài dùng để kiểm tra khả năng ghép nhiều phần của hệ thống thành một luồng nghiệp vụ có thể vận hành thật.

Đề bài: Learner Deadline Risk Dashboard

Xây một luồng cho admin/professor xem học viên có rủi ro trễ hạn.

Phạm vi

  1. Assignment-service phát event khi assignment được tạo/published.
  2. Deadline-service tạo reminder runs.
  3. Notification-service nhận reminder due và tạo inbox notification.
  4. Analytics-service consume submission/reminder events để tính risk.
  5. Backoffice-bff expose GET /api/backoffice/courses/{id}/deadline-risk.
  6. React admin hiển thị bảng risk, filter theo course/section/status.

Cách triển khai theo từng phần

  1. Bắt đầu ở backend: thêm command publish assignment và đảm bảo assignment có deadline, courseId, sectionId, phạm vi người học.
  2. Thêm outbox event assignment.published hoặc deadline.reminder.due; payload phải đủ để consumer xử lý mà không đọc DB của service phát event.
  3. Ở notification-service, tạo inbox item idempotent theo eventId. Gửi lại cùng event hai lần không được tạo hai notification.
  4. Ở analytics-service, tính risk đơn giản trước: quá hạn, gần đến hạn, chưa nộp, điểm quiz thấp. Sau đó mới mở rộng rule phức tạp hơn.
  5. Ở backoffice-bff, gom dữ liệu course, deadline và risk thành một DTO đúng nhu cầu màn hình. Không để React admin gọi nhiều service trực tiếp.
  6. Ở React admin, làm bảng có filter, empty state, loading state, error state và link về người học/course detail.
Không làm quá rộng ngay lần đầu. Phiên bản đầu chỉ cần một course, một section, một loại deadline và một công thức risk đơn giản. Khi luồng đã chạy đúng, mới thêm nhiều loại reminder hoặc rule phân tích nâng cao.

Kết quả cần đạt

  • Có thiết kế ngắn 1-2 trang: service ownership, API, event, DB, failure cases.
  • Có migration, DTO, service, repository, controller, consumer, BFF và UI.
  • Có test cho gateway/BFF hoặc consumer idempotency.
  • Có demo data profile local.
  • Có observability tối thiểu: correlation id/log rõ cho request và event.

Lỗi thường gặp và cách tránh

Lỗi Vì sao nguy hiểm Cách đúng
Frontend gọi thẳng port service Client dính vào topology microservice, bypass gateway security. Đi qua api-gateway, sau đó BFF hoặc public/internal route.
Service đọc DB service khác Phá DB-per-service, khó scale và đổi schema. Gọi API, consume event, hoặc dùng read model.
Nhét logic màn hình vào gateway Gateway phình thành BFF trá hình. Gateway mỏng; BFF compose data theo màn hình.
Gửi Kafka trực tiếp sau khi update DB Dễ dual-write: DB thành công nhưng Kafka fail hoặc ngược lại. Transactional outbox.
Consumer không dedup Kafka có thể gửi lại event, gây duplicate notification/grade/metric. Bảng processed_events trong transaction.
Demo data nằm trong schema changelog Production có thể bị insert data mẫu. Tách db.changelog-demo.yaml và profile demo.
