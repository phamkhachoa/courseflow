CourseFlow v2 - Hướng dẫn triển khai chi tiết

Tài liệu này hướng dẫn xây hệ thống theo thứ tự hợp lý: dựng khung, tách service nghiệp vụ, thêm feature thật, chuẩn hóa event reliability, rồi mới mở rộng search/analytics/client.

ĐÃ CÓ code chạy được trong repo MỘT PHẦN có khung/schema, còn thiếu logic CHƯA CÓ mới chỉ là tài liệu, intern phải tự viết

1. Nguyên tắc v2

  • Không còn core-service. Mỗi service sở hữu một bounded context.
  • Không service nào query database của service khác.
  • Common library không chứa nghiệp vụ LMS.
  • Event schema đặt ở event-contracts, không import entity của service khác.
  • Event producer dùng transactional outbox; consumer dùng dedup.
  • Next.js chỉ dùng cho learner/public/SEO; React chỉ dùng cho admin/backoffice.
  • api-gateway là trust boundary: verify JWT, strip X-User-* giả mạo và inject identity header đã xác thực.
  • Infra nằm trong từng repo folder, không trộn vào source service.

1b. Trạng thái code thật trong repo (đọc trước khi bắt đầu)

Tài liệu mô tả hệ thống đích. Code hiện tại trong repo chưa đạt tới đó. Bảng dưới đây nói rõ cái gì đã chạy, cái gì mới là khung, cái gì bạn phải tự viết — để bạn không tưởng nhầm là "chỉ cần ráp lại".

5. Phase mẫu hoàn chỉnh: Announcement → Notification ĐÃ IMPL TRONG REPO

Đây là flow event-driven end-to-end đầu tiên và đã được code thật trong repo (OutboxRelay.java ở announcement-service, AnnouncementEventConsumer.java ở notification-service). Nó là khuôn mẫu cho mọi flow khác (deadline→notification, quiz→gradebook, course→search, grade→analytics...). Producer của nhiều flow này đã ghi outbox sẵn — bạn chỉ cần copy đúng 2 mảnh còn thiếu: relay và consumer dedup.

Trạng thái kiểm chứng: code đã compile pass (mvn -pl services/announcement-service,services/notification-service -am compile). Chưa chạy end-to-end vì môi trường tạo tài liệu không có Docker để bật Kafka/Postgres. Phần "Definition of Done" bên dưới là các lệnh để bạn tự verify runtime. 
announcement-service                outbox-relay              notification-service
  publish()                          (poll mỗi 1s)              @KafkaListener
  ├─ UPDATE announcements            SELECT * FROM              ├─ check processed_events
  └─ INSERT outbox_events  ───────▶  outbox_events      ──────▶ ├─ INSERT notification
     (cùng 1 transaction)            WHERE published_at         └─ INSERT processed_events

Hai điểm lệch doc↔code đã được sửa trong repo (giữ lại đây để bạn hiểu vì sao):
  • Relay DB: workers/outbox-relay/application.yml trỏ vào một DB duy nhất cf_outbox, nhưng mỗi service ghi outbox_events vào DB riêng của nó (vd cf_announcement). Cách đã chọn cho training: relay nằm chung process với service phát event, đọc đúng DB của service đó (xem announcement-service/config/OutboxRelay.java). workers/outbox-relay để lại làm scaffold cho hướng Debezium/relay riêng sau này.
  • eventId trong payload: AnnouncementService.publish() nay sinh eventId và đưa vào payload outbox để consumer dùng làm khóa dedup. Trước đây payload thiếu field này nên dedup không chạy được.

Bước 0 — Bật Kafka cho đúng (đã có sẵn)

Dependency spring-kafka đã có trong announcement-service và notification-service. Kafka broker chạy ở localhost:9092 (xem docker-compose). Không cần thêm gì.

Bước 1 — Producer: sinh eventId trong outbox payload ĐÃ IMPL

File: announcement-service/.../service/AnnouncementService.java. Method publish() sinh eventId ổn định (dùng làm khóa dedup):

@Transactional
public AnnouncementDto publish(UUID announcementId) {
    announcements.publish(announcementId);
    AnnouncementDto published = get(announcementId);
    String eventId = UUID.randomUUID().toString();          // <-- đã thêm
    announcements.outbox(announcementId, "announcement.published", toJson(Map.of(
            "eventId", eventId,                              // <-- đã thêm
            "announcementId", announcementId.toString(),
            "courseId", published.courseId(),
            "title", published.title(),
            "audience", published.audience(),
            "publishedAt", published.publishedAt().toString())));
    return published;
}
Outbox INSERT và UPDATE announcement nằm trong cùng @Transactional → không bao giờ có chuyện "đã publish nhưng mất event" hay "có event nhưng chưa publish". Đây là điểm cốt lõi của outbox pattern.

Bước 2 — Relay: poll outbox và đẩy lên Kafka ĐÃ IMPL

File: announcement-service/.../config/OutboxRelay.java (relay đặt ngay trong service phát event, đọc đúng DB của service đó). Đã bật @EnableScheduling trên AnnouncementServiceApplication.

@Component
public class OutboxRelay {

    private final JdbcClient jdbc;
    private final KafkaTemplate<String, String> kafka;

    public OutboxRelay(JdbcClient jdbc, KafkaTemplate<String, String> kafka) {
        this.jdbc = jdbc;
        this.kafka = kafka;
    }

    @Scheduled(fixedDelayString = "${courseflow.outbox.poll-interval-ms:1000}")
    @Transactional
    public void relay() {
        var rows = jdbc.sql("""
                SELECT id, event_type, payload::text AS payload
                FROM outbox_events
                WHERE published_at IS NULL
                ORDER BY created_at
                LIMIT 100
                FOR UPDATE SKIP LOCKED
                """)
            .query((rs, n) -> Map.of(
                "id", rs.getObject("id", UUID.class),
                "eventType", rs.getString("event_type"),
                "payload", rs.getString("payload")))
            .list();

        for (var row : rows) {
            String topic = (String) row.get("eventType");          // "announcement.published"
            kafka.send(topic, (String) row.get("payload"))
                 .join();                                          // chờ ack trước khi mark
            jdbc.sql("UPDATE outbox_events SET published_at = NOW() WHERE id = :id")
                .param("id", row.get("id"))
                .update();
        }
    }
}

Bật scheduling trong AnnouncementServiceApplication:

@EnableScheduling
@SpringBootApplication(scanBasePackages = "edu.courseflow")
public class AnnouncementServiceApplication { ... }
FOR UPDATE SKIP LOCKED cho phép chạy nhiều instance relay mà không gửi trùng. kafka.send().join() đảm bảo chỉ mark published_at sau khi Kafka đã nhận — nếu gửi fail, row vẫn còn để lần sau retry (at-least-once). Vì vậy consumer bắt buộc phải dedup.

Bước 3 — Consumer: nhận event + dedup + tạo notification ĐÃ IMPL

File: notification-service/.../consumer/AnnouncementEventConsumer.java. Bảng processed_events đã có sẵn trong migration. 

@Component
public class AnnouncementEventConsumer {

    private static final String CONSUMER = "notification:announcement.published";
    private final JdbcClient jdbc;
    private final ObjectMapper objectMapper;

    public AnnouncementEventConsumer(JdbcClient jdbc, ObjectMapper objectMapper) {
        this.jdbc = jdbc;
        this.objectMapper = objectMapper;
    }

    @KafkaListener(topics = "announcement.published", groupId = "notification-service")
    @Transactional
    public void onAnnouncementPublished(String payload) throws Exception {
        JsonNode e = objectMapper.readTree(payload);
        UUID eventId = UUID.fromString(e.get("eventId").asText());

        // 1) Dedup: nếu đã xử lý eventId này rồi thì bỏ qua (idempotent)
        int inserted = jdbc.sql("""
                INSERT INTO processed_events (event_id, consumer_name)
                VALUES (:eventId, :consumer)
                ON CONFLICT (event_id) DO NOTHING
                """)
            .param("eventId", eventId)
            .param("consumer", CONSUMER)
            .update();
        if (inserted == 0) {
            return;                                  // đã xử lý → không tạo notification trùng
        }

        // 2) Business effect: tạo notification (ở đây fan-out đơn giản tới author;
        //    thực tế sẽ resolve audience qua enrollment API/read model)
        jdbc.sql("""
                INSERT INTO notifications (id, user_id, notification_type, title, body)
                VALUES (:id, :userId, 'ANNOUNCEMENT', :title, :body)
                """)
            .param("id", UUID.randomUUID())
            .param("userId", e.get("courseId").asText())   // placeholder audience
            .param("title", e.get("title").asText())
            .param("body", "New announcement in course " + e.get("courseId").asText())
            .update();
    }
}
Dedup-insert và business-insert nằm trong cùng một transaction. Nếu tạo notification fail, cả processed_events cũng rollback → lần redeliver sau sẽ xử lý lại. Nếu tách 2 transaction, bạn sẽ có bug "đã đánh dấu processed nhưng notification chưa tạo".

Cấu hình consumer trong notification-service/application.yml:

spring:
  kafka:
    bootstrap-servers: ${KAFKA_BOOTSTRAP_SERVERS:localhost:9092}
    consumer:
      group-id: notification-service
      auto-offset-reset: earliest
      enable-auto-commit: false
    listener:
      ack-mode: record

Phase 3 - Enrollment service ĐÃ CÓ

enrollment-service sở hữu roster, waitlist, enrollment status và capacity policy.

Phase 9 - Outbox/Dedup chuẩn hóa CÓ FLOW MẪU · CẦN NHÂN BẢN

Code mẫu đầy đủ (relay + consumer + dedup) đã impl cho flow announcement→notification — xem mục Phase mẫu. Việc còn lại: nhân bản relay/consumer cho các flow producer khác đã có sẵn outbox.

Đặt phase này sau khi đã có event thật từ Announcement, Deadline, Assignment, Portfolio và Discussion.

Producer pattern

@Transactional
businessUseCase():
  save domain change
  save outbox_events(eventId, aggregateId, eventType, payload)

Debezium hoặc outbox-relay:
  publish event to Kafka
  mark publishedAt nếu dùng relay

Consumer pattern

onEvent(event):
  if processed_events contains eventId + consumerName:
    return
  process business effect
  insert processed_events
Không gửi Kafka trực tiếp trong transaction nghiệp vụ nếu chưa có outbox. Đó là dual-write anti-pattern.

Module Ownership

Phase 14 - Quiz + Question Bank CRUD + OUTBOX ĐÃ CÓ · CONSUMER CHƯA CÓ

Quiz là feature LMS thật và rất tốt cho training vì có modeling phức tạp, timed attempt, idempotent submit, auto grading và tích hợp gradebook.

Service owner

quiz-service sở hữu question bank, quiz definition, quiz attempt, answer snapshot và auto-grade result.

Schema gợi ý

question_banks(id, course_id, title)
questions(id, bank_id, type, stem, difficulty, points, status)
question_options(id, question_id, label, content, correct)
quizzes(id, course_id, title, open_at, close_at, duration_minutes, attempts_allowed)
quiz_questions(id, quiz_id, question_id, points, position)
quiz_attempts(id, quiz_id, student_id, attempt_no, started_at, submitted_at, status, score)
quiz_answers(id, attempt_id, question_id, answer_payload, score, graded_at)

Implementation steps

  1. Professor tạo question bank và câu hỏi: multiple choice, true/false, short answer.
  2. Tạo quiz từ bank, chọn policy: duration, attempts allowed, randomize questions/options.
  3. Student start attempt: server tạo snapshot câu hỏi để không bị thay đổi giữa lúc làm.
  4. Student submit: API idempotent theo attemptId, auto-grade câu hỏi khách quan.
  5. Publish quiz.attempt.submitted và quiz.attempt.graded.
  6. Gradebook-service tạo/update grade item tương ứng.

Checklist pass

  • Submit lại cùng attempt không tạo điểm trùng.
  • Hết giờ thì attempt chuyển trạng thái đúng.
  • Question snapshot giữ nguyên dù professor sửa bank.
  • Gradebook nhận điểm quiz và tính vào final score.

Phase 16 - Certificate Verification CRUD + OUTBOX ĐÃ CÓ · PDF/CONSUMER CHƯA CÓ

Certificate Verification là use case rất hợp cho Next.js: public, SEO-friendly, shareable và cần bảo mật. Đây cũng là feature nối trực tiếp với Gradebook vì phải dựa trên final grade/course completion.

Service owner

certificate-service sở hữu certificate issue, verification code, status, audit và PDF metadata.

Schema gợi ý

certificates(id, student_id, course_id, final_grade, issued_at, expires_at, status)
certificate_verifications(id, certificate_id, verification_code, signature, public_slug)
certificate_audit_logs(id, certificate_id, action, actor_id, reason, created_at)

Implementation steps

  1. Consume gradebook.final_grade.updated.
  2. Kiểm tra completion rule: đủ module, pass minimum score, không bị hold.
  3. Issue certificate với verification code không đoán được.
  4. Tạo public route Next.js: /certificates/verify/[code].
  5. Thêm revoke/reissue workflow trong React admin.

Security checklist

  • Verification code không chứa user id/course id thô.
  • Public API chỉ trả dữ liệu được phép công khai.
  • Certificate revoked không còn pass verify.
  • Reissue/revoke có audit log.

Phase 17 - Peer Review / Workshop CRUD + OUTBOX ĐÃ CÓ · CONSUMER CHƯA CÓ

Peer Review là feature workflow nâng cao: nhiều bước, nhiều actor, deadline riêng, anonymous policy, dispute và finalize grade. Đây là bài training rất tốt cho state machine và event orchestration.

Service owner

peer-review-service sở hữu reviewer assignment, review form, review submission, dispute và finalize workflow.

Schema gợi ý

peer_review_settings(id, assignment_id, reviewers_per_submission, anonymous, review_due_at)
review_assignments(id, assignment_id, submission_id, reviewer_id, status)
review_forms(id, assignment_id, rubric_template_id, instructions)
review_submissions(id, review_assignment_id, score, comment, submitted_at, status)
review_disputes(id, review_submission_id, opened_by, reason, status)
peer_review_results(id, submission_id, final_score, finalized_by, finalized_at)

Implementation steps

  1. Professor bật peer review cho assignment và chọn rubric.
  2. Sau deadline nộp bài, service tự assign reviewer theo policy.
  3. Notification-service gửi peer_review.assigned.
  4. Student review bài được giao; reviewer có thể anonymous.
  5. Professor/TA xử lý dispute và finalize score.
  6. Publish peer_review.finalized cho gradebook-service.

Checklist pass

  • Reviewer không được review bài của chính mình.
  • Anonymous mode không leak reviewer identity cho student.
  • Late review được đánh dấu đúng và notification nhắc deadline.
  • Finalized score sync sang Gradebook, không tạo grade trùng.

Testing, CI/CD và hardening

Layer Kiểm thử
Unit Service use cases, validation, policy.
Integration PostgreSQL/Mongo/Redis/Kafka bằng Testcontainers.
Contract BFF ↔ service APIs, event schema compatibility.
E2E Course publish -> search -> enroll -> module progress -> assignment/quiz -> gradebook -> certificate/peer review -> analytics.
Observability Correlation ID, metrics, logs, traces, dashboards.

Production checklist

  • Every service has health/info/metrics endpoints.
  • Every service has migration and seed data for local test.
  • Every event producer uses outbox.
  • Every event consumer uses dedup.
  • No service queries another service database.
  • Frontend routes map to real use cases, not technology demos.
  • Advanced features publish events into Gradebook/Analytics instead of recalculating across databases.
