CourseFlow v2 - identity-service

Identity Service đang làm gì?

Identity-service là service sở hữu user, credential, token và RBAC. Gateway đứng ở biên để xác thực access token, còn quyết định quyền chi tiết được thiết kế để hỏi lại identity qua authz endpoint khi domain service cần.

Authentication: xác minh bạn là ai Authorization: bạn được làm gì RBAC: role, permission, scope
Public
/api/auth/login, /api/auth/refresh Đi qua gateway tới identity. Login kiểm tra password, refresh xoay vòng refresh token.
Protected
/api/internal/**, /api/learning/** Cần Bearer JWT hợp lệ. Gateway inject X-User-* cho service phía sau.
Backoffice
/api/backoffice/**, /api/users/** Cần role operator ở gateway: ADMIN, ORG_ADMIN, INSTRUCTOR, PROFESSOR hoặc TA.
Authz
/internal/authz/check Identity kiểm tra permission theo role hierarchy, scope và rule DENY thắng ALLOW.
JWT Access token ngắn hạn, có claim uid, sub email và danh sách role.
Refresh Refresh token là chuỗi opaque, DB chỉ lưu SHA-256 hash, refresh thì revoke token cũ.
Headers Gateway tự gắn X-User-Id, X-User-Role, X-User-Roles, X-User-Email.
RBAC Role có thể kế thừa, permission có ALLOW/DENY, assignment có scope PLATFORM/ORG/COURSE/DEPARTMENT.

Identity-Service Sở Hữu Những Gì

Theo architecture, identity-service chỉ sở hữu danh tính, credential và quyền. Course, enrollment, assignment và các business rule khác thuộc service riêng.

User & Credential

Quản lý bảng users: email, BCrypt password hash, full name, role mặc định và status. API quản trị user nằm ở /backoffice/users.

Token Lifecycle

/auth/login phát access token và refresh token. /auth/refresh rotate refresh token. /auth/logout revoke refresh token theo user hiện tại.

RBAC Engine

Quản lý roles, permissions, role_permission_grants và user_role_assignments. Endpoint chính: /internal/authz/check.

Luồng Authentication

Authentication trả lời câu hỏi: request này đến từ user nào, token có thật và còn hạn không?

1

Client đăng nhập

React admin hoặc Flutter gọi POST /api/auth/login với email và password.

2

Identity kiểm credential

AuthService.login tìm user ACTIVE theo email và dùng BCrypt để match password.

3

Phát token

JwtTokenProvider tạo access JWT. RefreshTokenService tạo refresh token opaque và lưu hash.

4

Client lưu token

React admin lưu localStorage. Flutter lưu bằng secure storage Keychain/Keystore.

5

Request sau đó

Client gắn Authorization: Bearer <accessToken> cho mọi API cần đăng nhập.

Một Request Protected Đi Qua Những Chặng Nào

Ví dụ: client gọi POST /api/internal/assignments hoặc một API nội bộ khác.

Client Bearer access token
›
API Gateway Verify JWT, harden headers
›
BFF / Domain Đọc identity headers
›
Identity Authz Check permission khi cần
Client gửi request

Request có header Authorization: Bearer .... Nếu client tự gửi X-User-*, các header đó không được tin.

Gateway xóa identity giả mạo

JwtAuthenticationGatewayFilter luôn remove X-User-Id, X-User-Role, X-User-Roles, X-User-Email trước khi xử lý.

Gateway verify JWT

Gateway dùng shared secret với identity-service để verify chữ ký và hạn token. Thiếu token, token sai hoặc expired sẽ trả 401 Unauthorized.

Gateway extract claims

Lấy uid, email từ subject, và roles. Role chính được chọn theo thứ tự ưu tiên: ADMIN, ORG_ADMIN, INSTRUCTOR, PROFESSOR, TA, STUDENT.

Gateway chặn backoffice nếu thiếu role

Với /api/backoffice/** hoặc /api/users/**, user không có role operator sẽ bị 403 Forbidden.

Gateway inject header tin cậy

Downstream nhận X-User-Id, X-User-Email, X-User-Role, X-User-Roles. Common library có CurrentUserArgumentResolver để parse các header này.

Domain service xử lý authorization

Với check đơn giản, service có thể đọc role từ header. Với quyền mịn như course:publish hoặc gradebook:manage, service nên gọi POST /internal/authz/check của identity.

Luồng Authorization / Authz

Authorization trả lời câu hỏi: user này có quyền thực hiện action này trong scope này không?

Payload check quyền

Domain service gửi request kiểu:

{
  "userId": "4",
  "permission": "course:publish",
  "scopeType": "ORG",
  "scopeId": "org-42"
}

Identity trả về allowed: true hoặc false.

Cách identity quyết định

  • Lấy role assignment còn hiệu lực của user ở PLATFORM hoặc đúng scope request.
  • Đi theo role hierarchy bằng recursive query, ví dụ TA kế thừa STUDENT.
  • Tìm grant cho permission trong role_permission_grants.
  • Nếu có DENY thì deny ngay, kể cả có ALLOW ở role khác.
  • ADMIN platform được allow mọi thứ nếu không có explicit DENY.
  • Mặc định không có grant thì deny.

Phân Quyền Nên Đặt Ở Gateway Hay Từng Service?

Kết luận phù hợp với kiến trúc hiện tại: gateway chỉ nên làm lớp bảo vệ biên và phân quyền thô; quyết định quyền cuối cùng phải nằm ở BFF/domain service, còn identity-service là nơi tập trung policy RBAC.

Cách nên làm

Gateway = authentication + coarse authorization. Service API = business/resource authorization. Identity-service = RBAC policy engine.

Cách không nên làm

Không dồn toàn bộ authorization vào gateway, vì gateway không sở hữu dữ liệu business như owner course, enrollment, submission, gradebook scope hoặc trạng thái workflow.

Layer Nên chịu trách nhiệm Không nên chịu trách nhiệm
API Gateway Verify JWT, strip header giả, inject X-User-*, phân biệt public/protected, chặn route thô như backoffice cần operator role. Không quyết định các quyền cần dữ liệu domain: user có dạy course này không, student có enrolled không, submission có thuộc user này không.
BFF Guard theo màn hình hoặc workflow để trả lỗi sớm, ví dụ backoffice screen chỉ cho role operator. Không là lớp bảo vệ duy nhất, vì domain API vẫn có thể được gọi bởi BFF khác hoặc service-to-service.
Domain Service Enforce quyền ở từng endpoint/use case, kiểm tra owner, membership, enrollment, trạng thái dữ liệu và gọi identity authz khi cần permission mịn. Không tự định nghĩa role/permission lệch khỏi identity-service; tránh mỗi service có một policy riêng khó kiểm soát.
Identity-Service Là source of truth cho user, role, permission, assignment, scope và endpoint /internal/authz/check. Không nên chứa business rule của domain như publish course cần draft đã review hay gradebook đã locked chưa.

Vì sao không để hết ở gateway?

  • Gateway chỉ nhìn path, method, JWT claim; nó không biết dữ liệu chi tiết trong database của từng service.
  • Object-level authorization cần context: course owner, org scope, enrollment status, assignment due date, submission owner.
  • Rule business thay đổi theo domain; nếu nhét vào gateway thì gateway phình to và coupling với mọi service.
  • Service-to-service hoặc BFF khác gọi API vẫn cần domain service tự bảo vệ.
  • Gateway chỉ check role coarse dễ bị thiếu case, ví dụ INSTRUCTOR không phải lúc nào cũng được sửa mọi course.

Vì sao không bỏ gateway authz hoàn toàn?

  • Gateway là trust boundary, nên nó phải reject request không có token trước khi vào hệ thống.
  • Các route rõ ràng như /api/backoffice/** nên bị chặn sớm nếu là STUDENT.
  • Strip header giả mạo ở gateway giúp service phía sau có thể tin X-User-*.
  • Chặn thô ở gateway giảm tải cho service và làm lỗi 401/403 nhất quán hơn.
API ví dụ Gateway nên làm Service nên làm
POST /api/auth/login Cho qua public route tới identity-service. Identity kiểm email, password hash và status ACTIVE.
GET /api/backoffice/users Yêu cầu JWT hợp lệ và role operator. Identity filter vẫn kiểm tra có gateway identity và role backoffice để phòng direct access.
POST /api/internal/courses/{id}/publish Yêu cầu authenticated request. Course-service kiểm user có course:publish trong scope phù hợp và business state của draft.
POST /api/internal/gradebook/entries Yêu cầu authenticated request. Gradebook-service kiểm gradebook:manage, course scope và policy gradebook như lock/finalized.
POST /api/internal/assignments/{id}/submissions Yêu cầu authenticated request. Assignment-service kiểm student là người nộp bài hợp lệ, đã enrolled, submission thuộc user và còn trong deadline.
Khuyến nghị implement tiếp trong repo này

Giữ gateway mỏng như tài liệu architecture đang định hướng, thêm route gateway cho identity authz endpoints, rồi thêm một AuthzClient dùng chung để domain service gọi /internal/authz/check. Ở controller/use case, lấy caller từ CurrentUser; không lấy user hiện tại từ request body do client gửi.

Model RBAC Hiện Có

Đây là các khái niệm chính đang có trong migration và service code.

Thành phần Ý nghĩa Ví dụ / file liên quan
Role Nhóm quyền có tên, có thể là system role hoặc role động do admin tạo. STUDENT, TA, INSTRUCTOR, ORG_ADMIN, ADMIN
Permission Quyền mịn theo format resource:action. course:publish, quiz:grade, user:manage
Grant Role được ALLOW hoặc DENY một permission. role_permission_grants(effect)
Assignment User được gán role trong scope cụ thể, có thể hết hạn. user_role_assignments(scope_type, scope_id, expires_at)
Scope Phạm vi áp dụng quyền. PLATFORM, ORG, COURSE, DEPARTMENT

Refresh Token & Logout

Client đã có logic tự refresh khi gặp 401, cả React admin và Flutter đều dùng single-flight để nhiều request lỗi cùng lúc chỉ refresh một lần.

Khi access token hết hạn

  1. API protected trả 401.
  2. Interceptor gọi POST /api/auth/refresh với refresh token.
  3. Identity hash refresh token, tìm token còn usable, revoke token cũ.
  4. Identity phát access token mới và refresh token mới.
  5. Client lưu token mới và retry request ban đầu một lần.

Khi logout

Client gọi POST /api/auth/logout. Gateway yêu cầu Bearer token rồi inject user header. Identity gọi revokeAll(userId) để vô hiệu refresh token của user, sau đó client xóa session local.

Trạng Thái Implement Hiện Tại

Các điểm này quan trọng nếu đang review hoặc chuẩn bị hardening security.

Đã enforce trong code

Gateway verify JWT, strip spoofed identity headers, inject header tin cậy và chặn backoffice/user-management theo role. Identity-service cũng có filter yêu cầu gateway identity cho endpoint không public.

Chưa thấy enforce đầy đủ ở domain controllers

Nhiều controller domain hiện chưa nhận CurrentUser và chưa gọi /internal/authz/check. Vì vậy fine-grained authorization đã có engine ở identity nhưng cần tích hợp vào từng domain service/BFF.

Gateway thiếu route cho identity internal authz

api-gateway/application.yml có route /api/auth/** và /api/users/** về identity, nhưng chưa có route cho /api/internal/roles, /api/internal/permissions, /api/internal/authz/check. React admin roles API đang gọi các path này qua gateway.

User mới cần role assignment

UserService.create đang tạo users.role; token provider có fallback role nếu chưa có assignment. Nhưng AuthzService.check đọc từ user_role_assignments, nên user mới cần được gán assignment để permission check mịn không bị deny mặc định.

401 Hay 403 Xảy Ra Khi Nào?

Phân biệt nhanh lỗi xác thực và lỗi phân quyền.

Status Ý nghĩa Tình huống trong code
401 Unauthorized Không xác thực được user. Thiếu Bearer token, token sai/expired, JWT thiếu uid, email hoặc không có role.
403 Forbidden Biết user là ai nhưng user không đủ quyền. Student vào /api/backoffice/** hoặc domain/authz check trả allowed=false.
